== Scope

=== In Scope

* ReScript parser sources and tests.
* Idris2 ABI package and checked proof core.
* Zig FFI runtime in `ffi/zig`.
* Example inputs, aspect tests, and benchmark surfaces.
* Public claim surfaces in `README.adoc`, `LEVEL-STATUS.md`, and paper drafts.

=== Out of Scope

* Container, devcontainer, Nix, and Guix packaging surfaces removed from the repo.
* Hosted service operations; this repo does not currently ship a maintained network service.
* Physical security and forge-platform internals.

== System Overview

`typed-wasm` is a research repository combining a ReScript parser, an Idris2
ABI/proof layer, and a Zig FFI runtime. The main security concern is claim
integrity and artifact integrity: the code, proofs, tests, and papers must all
describe the same system.

See link:../STATE-VISUALIZER.adoc[STATE-VISUALIZER.adoc] for the current repo
topology.

== Main Threats

=== Spoofing

* Forged maintainer or bot actions could smuggle false changes into public
  claims or workflows.

=== Tampering

* Parser, proof, and paper drift can create false claims of verification.
* Example inputs or benchmark scripts can be quietly altered to make evidence
  look stronger than it is.

=== Repudiation

* If audit notes and release metadata are not kept current, it becomes hard to
  justify why a claim was made at a given time.

=== Information Disclosure

* Repository metadata or workflows could leak secrets if future automation is
  added carelessly.

=== Denial of Service

* Pathological example inputs or CI churn can waste maintainer time and mask
  real regressions.

=== Elevation of Privilege

* Future workflow or release automation could gain broader permissions than the
  repo actually needs.

== Mitigations In Place

* Checked claim envelope explicitly limited to the L1-L10 core.
* Aspect testing for README/manifest/roadmap coherence.
* Parser tests, end-to-end smoke tests, and benchmark surface present.
* Template release and packaging scaffolds removed rather than left as false evidence.
* Workflow pinning and audit gating present at the standards level.

== Residual Risks

* L11-L12 drafts remain in the tree and can still tempt overclaiming if the
  paper surface is not kept disciplined.
* ReScript warnings remain unresolved and should be treated as real technical debt.
* Venue-facing papers still need a dedicated theorem/benchmark parity review.

== Review Trigger

Review this threat model:

* before any `beta`, `stable`, or venue-facing claim
* after any change to the proof envelope
* after adding new release, packaging, or automation surfaces
